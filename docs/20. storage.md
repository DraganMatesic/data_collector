# Storage Management

**Implementation Status:** IMPLEMENTED | **Work Package:** WP-13 | **Milestone:** v0.5.0

## Overview

File and blob storage management for downloaded documents, images, PDFs, and extracted data. Provides consistent directory organization, SHA-256 hash-based deduplication, per-file retention classification, pluggable storage backends, and multi-backend file transfer. Developed and tested primarily on Windows, but fully deployable on Linux.

## Architecture

```
StorageManager
├── BaseStorageBackend (ABC)
│   └── FilesystemBackend (local, UNC, mounted shares)
├── StoredFile (ORM table -- centralized metadata)
├── CodebookFileRetention (retention categories)
├── RetentionPolicy (filesystem-based enforcement)
└── enforce_retention_by_database() (DB-driven enforcement)
```

## Storage Backends

| Backend | Use Case | Status |
|---------|----------|--------|
| Local filesystem | Development, single-server | Implemented |
| Network shares (UNC/mounts) | Multi-server, shared storage | Implemented |
| Cloud storage (S3, GCS) | Scalable, managed | Planned |

### BaseStorageBackend Interface

All backends implement the `BaseStorageBackend` ABC:

```python
from data_collector.storage import BaseStorageBackend, FilesystemBackend

# Local backend
local = FilesystemBackend(Path("C:/storage"), location="local")

# Network share backend
remote = FilesystemBackend(Path("//fileserver/share"), location="fs_market_hr")
```

Methods: `store()`, `retrieve()`, `delete()`, `exists()`, `get_size()`.

### Backend Registry

The `storage_backend` configuration table serves as a central registry of named backends. DBAs add or modify backends at runtime without code changes or redeployment.

| Column | Type | Description |
|--------|------|-------------|
| `id` | auto_increment | Primary key |
| `location_name` | String(128), unique | Backend identifier (e.g., ``local``, ``fs_market_hr``) |
| `root_path` | UnicodeText | Root directory (local path, UNC path, or mount point) |
| `description` | String(256) | Human-readable description |
| `is_active` | Boolean | Enable/disable without deleting |
| `min_free_disk_gb` | Float, nullable | Per-backend disk free space threshold (GB). NULL = use global default |
| `max_storage_alert_gb` | Float, nullable | Per-backend storage budget (GB). NULL = use global default or disabled |
| `date_created` | DateTime | Row creation timestamp |

**Adding a backend (DBA):**

```sql
INSERT INTO storage_backend (location_name, root_path, description)
VALUES ('fs_market_hr', '\\fileserver\hr_share', 'Croatia market file server');

INSERT INTO storage_backend (location_name, root_path, description)
VALUES ('fs_market_de', '\\fileserver\de_share', 'Germany market file server');
```

**Using a named backend (application code):**

```python
# Resolve by name from database -- no hardcoded paths
manager = StorageManager(database, "hr", "registry", "companies",
    runtime_id=runtime_id, backend="fs_market_hr")

# Or resolve explicitly for transfer targets
remote = StorageManager.resolve_backend(database, "fs_market_de", session)
manager.transfer(stored_file, remote, delete_source=True, session=session)
```

When no backend is specified, `StorageManager` falls back to `StorageSettings.root` as the default local backend (zero-config).

## File Organization

### Directory Structure Convention

All paths use `pathlib.Path` for cross-platform compatibility (Windows + Linux). Files are organized by extension, then by configurable date depth:

```
storage_root/
├── {group}/
│   └── {parent}/
│       └── {app_name}/
│           ├── pdf/
│           │   └── {YYYY-MM-DD}/
│           │       └── {sha256_hash}.pdf
│           └── csv/
│               └── {YYYY-MM-DD}/
│                   └── {sha256_hash}.csv
```

**Examples:**
- Windows: `C:\storage\hr\registry\company_scraper\pdf\2025-01-15\`
- Linux: `/opt/storage/hr/registry/company_scraper/csv/2025-01-15/`

### Date Depth

The date folder granularity is controlled by `DC_STORAGE_DIRECTORY_DEPTH`:

| Depth | Format | Example Path |
|-------|--------|-------------|
| `flat` | (none) | `app_name/pdf/{hash}.pdf` |
| `yearly` | `YYYY` | `app_name/pdf/2025/{hash}.pdf` |
| `monthly` | `YYYY-MM` | `app_name/pdf/2025-01/{hash}.pdf` |
| `daily` | `YYYY-MM-DD` | `app_name/pdf/2025-01-15/{hash}.pdf` |
| `hourly` | `YYYY-MM-DD-HH` | `app_name/pdf/2025-01-15-14/{hash}.pdf` |

Default is `daily`. Use `flat` when file volumes are low, `hourly` for high-throughput ingestion.

### File Naming

Files are named by their SHA-256 content hash: `{sha256_hex}.{ext}`. The original filename is preserved in the `StoredFile` database record.

## Retention Policies

Retention is managed per-file via the `CodebookFileRetention` codebook table. The database is the source of truth for retention periods -- no retention days are hardcoded. Companies can modify default values or add custom categories by inserting rows into the codebook table.

### Default Categories

| Category | Enum Value | Retention Days | Use Cases |
|----------|-----------|---------------|-----------|
| TRANSIENT | 1 | 7 | Temporary files, caches, intermediate processing outputs |
| SHORT_TERM | 2 | 90 | Operational data, session files, monitoring snapshots |
| STANDARD | 3 | 365 | General business records, routine correspondence |
| REGULATORY_3Y | 4 | 1095 | Employment records, customer complaints, warranty documentation |
| REGULATORY_5Y | 5 | 1825 | Accounting records, financial statements, contractual documents |
| REGULATORY_7Y | 6 | 2555 | Tax and audit records, securities compliance, AML records |
| REGULATORY_10Y | 7 | 3650 | Banking records, insurance documents, healthcare records |
| EXTENDED | 8 | 9125 | Legal holds, product liability, engineering records |
| PERMANENT | 9 | NULL | Permanent storage, never deleted by retention enforcement |

### Extensibility

Companies can add custom retention categories post-deployment:

```sql
-- Adjust existing tier for local regulation
UPDATE c_file_retention SET retention_days = 4015 WHERE id = 7;  -- 11 years for Croatian tax

-- Add custom category
INSERT INTO c_file_retention (id, description, retention_days)
VALUES (10, 'Custom audit 15Y', 5475);
```

Custom categories are used by passing the integer ID to `store()`:

```python
manager.store(content, "pdf", original_filename="tax.pdf", retention_category=10, session=session)
```

Each file's `expiration_date` is computed at store time from `CodebookFileRetention.retention_days`. The `enforce_retention_by_database()` function queries expired rows and deletes both the physical file and the database record.

## Database Tables

### StoredFile

Centralized metadata for all managed files:

| Column | Type | Description |
|--------|------|-------------|
| `id` | auto_increment | Primary key |
| `app_id` | String(64) | Application identifier |
| `runtime` | String(64) | Runtime session identifier |
| `content_hash` | String(64) | SHA-256 hex digest of file content |
| `original_filename` | UnicodeText | Original filename before hash renaming |
| `stored_path` | UnicodeText | Relative path from backend root |
| `file_extension` | String(32) | File extension without leading dot |
| `file_size` | BigInteger | File size in bytes |
| `location` | String(256) | Backend identifier (e.g., "local", "fs_market_hr") |
| `retention_category` | BigInteger | FK to c_file_retention.id |
| `expiration_date` | DateTime | Computed expiration, NULL for permanent |
| `date_created` | DateTime | Row creation timestamp |

### CodebookFileRetention

Codebook for retention categories, seeded from `FileRetention` enum:

| Column | Type | Description |
|--------|------|-------------|
| `id` | BigInteger | PK from FileRetention enum |
| `description` | String(128) | Category description |
| `retention_days` | BigInteger | Days to retain, NULL for permanent |
| `sha` | String(64) | Hash for merge-based seeding |
| `archive` | DateTime | Soft delete timestamp |
| `date_created` | DateTime | Row creation timestamp |

## Usage

### Basic Storage

```python
from data_collector.enums.storage import FileRetention
from data_collector.storage import StorageManager

manager = StorageManager(
    database, "hr", "registry", "companies",
    runtime_id=runtime_id,
)

# Store file content (defaults to STANDARD retention)
path = manager.store(
    pdf_bytes, "pdf",
    original_filename="annual_report_2025.pdf",
    session=session,
)

# Store with specific retention category
path = manager.store(
    pdf_bytes, "pdf",
    original_filename="tax_filing.pdf",
    retention_category=FileRetention.REGULATORY_7Y,
    session=session,
)

# Store with custom company category (integer ID)
path = manager.store(
    pdf_bytes, "pdf",
    original_filename="audit_report.pdf",
    retention_category=10,
    session=session,
)

# Store from existing file
path = manager.store_file(
    Path("/downloads/report.pdf"),
    session=session,
)
```

### File Transfer

```python
from data_collector.storage import FilesystemBackend

remote = FilesystemBackend(Path("//fileserver/share"), location="fs_hr_prod")

# Transfer and delete local copy
manager.transfer(stored_file, remote, delete_source=True, session=session)

# Transfer and keep local copy
manager.transfer(stored_file, remote, delete_source=False, session=session)
```

### Retention Enforcement

```python
# DB-driven retention (primary mechanism)
deleted_count = manager.enforce_retention(session=session)

# Filesystem-based retention (by file age)
from data_collector.storage import enforce_retention, RetentionPolicy

policy = RetentionPolicy(max_age_days=90)
deleted_count = enforce_retention(backend, policy)
```

### Querying

```python
# List stored files
files = manager.get_stored_files(session=session)
files_on_date = manager.get_stored_files(session=session, date="2025-01-15")

# Storage usage
total_bytes = manager.get_storage_size(session=session)
```

## Configuration

Environment variables with `DC_STORAGE_` prefix:

| Setting | Default | Description |
|---------|---------|-------------|
| `DC_STORAGE_ROOT` | `./storage` | Base directory for file storage |
| `DC_STORAGE_DEDUPLICATE` | `true` | Enable SHA-256 hash-based deduplication |
| `DC_STORAGE_DIRECTORY_DEPTH` | `daily` | Date folder granularity: flat, yearly, monthly, daily, hourly |
| `DC_STORAGE_DEFAULT_RETENTION` | `standard` | Default retention category name |
| `DC_STORAGE_MIN_FREE_DISK_GB` | `10.0` | Alert when disk free space drops below (GB). Global default, overridden per-backend |
| `DC_STORAGE_MAX_STORAGE_ALERT_GB` | (none) | Optional: alert when per-backend stored data exceeds (GB). Disabled when unset |

## Manager Integration (StorageJanitor)

The Manager composes a ``StorageJanitor`` subsystem that runs periodically within the main tick loop. It iterates all active backends from the ``StorageBackend`` table and performs three tasks:

1. **Retention enforcement** -- deletes expired files and their ``StoredFile`` rows via ``enforce_retention_by_database()``
2. **Disk free space monitoring** (mandatory) -- checks physical disk free space via ``shutil.disk_usage()``
3. **Storage budget monitoring** (optional) -- computes total stored bytes per backend and compares against budget

Alert thresholds are resolved per-backend: each ``StorageBackend`` row can override the global defaults from ``StorageSettings``.  This allows different thresholds for a 256 GB local SSD vs. a 10 TB file server.

StorageJanitor is completely isolated from other Manager subsystems. Any exception during maintenance is caught and logged -- it never affects app scheduling, command dispatch, or process monitoring.

| Setting | Default | Description |
|---------|---------|-------------|
| `DC_MANAGER_STORAGE_JANITOR_ENABLED` | `true` | Enable periodic storage maintenance |
| `DC_MANAGER_STORAGE_JANITOR_CHECK_INTERVAL` | `3600` | Seconds between maintenance runs |
