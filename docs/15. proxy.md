# Proxy Management

**Implementation Status:** IN DEVELOPMENT | **Work Package:** WP-05 | **Milestone:** v0.3.0

## Overview

Proxy management for web scraping with IP verification and domain-level reservation. The framework integrates with commercial proxy services (BrightData, Oxylabs, Smartproxy, NetNut) that handle IP selection and rotation automatically via super-proxy gateways. The framework's responsibility is:

1. **Build proxy URLs** via a pluggable `ProxyProvider` interface
2. **Check provider health** via `ProxyProvider.is_healthy()` before acquisition
3. **Verify the assigned IP** via proxy judges with failover
4. **Check the IP blacklist** to skip known-bad IPs before reserving
5. **Reserve the IP** in the database so apps hitting the same root domain don't collide
6. **Release the reservation** on thread/session completion, with TTL-based auto-expiry for crash safety
7. **Report failures** via `ProxyManager.report_failure()` for incremental blacklist lockout

Built from the ground up based on enterprise experience with residential proxy services and government portal scraping across multiple countries.

**How it works:**

1. `ProxyProvider.is_healthy()` checks provider availability before starting
2. `ProxyManager.acquire()` generates a new session ID
3. `ProxyProvider.build_proxy_url(session_id)` constructs the provider-specific proxy URL
4. A proxy judge verifies the actual IP address assigned by the provider
5. The IP is checked against the `ProxyBlacklist` table for the target subdomain
6. The IP is checked against the `ProxyReservation` table for the root domain
7. If available, the IP is reserved; if not, a new session is tried
8. On completion, `ProxyManager.release()` marks the reservation as released
9. On failure (403, CAPTCHA, etc.), `ProxyManager.report_failure()` records the IP in the blacklist
10. Stale reservations (crashed apps) auto-expire via TTL

## ProxyProvider Interface

All major residential proxy services (BrightData, Oxylabs, Smartproxy, NetNut) use the same super-proxy gateway model — you connect to a single endpoint and the provider assigns an IP. The only difference between providers is the proxy URL format. The `ProxyProvider` interface abstracts this:

```python
from abc import ABC, abstractmethod

class ProxyProvider(ABC):
    """Interface for proxy service providers."""

    def __init__(self, proxy_data: ProxyData):
        self.proxy_data = proxy_data

    @abstractmethod
    def build_proxy_url(self, session_id: str) -> str:
        """Build proxy URL string for a given session ID.

        Args:
            session_id: Unique session identifier for IP pinning.

        Returns:
            Proxy URL string (e.g., "http://user:pass@host:port").
        """
        ...

    def is_healthy(self) -> bool:
        """Check if the proxy provider infrastructure is available.

        Default implementation builds a test proxy URL and verifies IP
        via the first proxy judge. Subclasses can override with
        provider-specific health APIs (e.g., BrightData status endpoint).

        Returns:
            True if a request through the proxy succeeds.
        """
        ...
```

Every provider receives the same `ProxyData` and uses it to construct provider-specific URLs. Adding a new field (e.g., `zone` for BrightData zones) means adding one field to `ProxyData` — no constructor signature changes across providers.

## IP Verification (Proxy Judges)

After the provider assigns an IP, the framework verifies it via proxy judges — lightweight endpoints that return your IP address. Multiple judges with failover ensure verification works even if one judge is down:

```python
PROXY_JUDGES = [
    "https://httpbin.org/ip",
    "https://api.ipify.org?format=json",
    "https://ifconfig.me/ip"
]
```

The verification flow:

1. Make a request through the proxy to the first judge URL
2. Parse the IP from the response
3. Sanity check: response content length < 365 bytes (prevents judge returning HTML error pages)
4. If the judge fails (timeout, error, invalid response), try the next judge
5. If all judges fail, retry with a new proxy session

```python
def verify_ip(proxy_url: str, judges: list[str]) -> str | None:
    """Try proxy judges with failover. Return IP string or None."""
    request = Request(timeout=10, retries=1)
    request.set_proxy(proxy_url)
    for judge_url in judges:
        try:
            response = request.get(judge_url)
            if response is None:
                continue
            if len(response.content) < 365:
                return extract_ip(response.content)
        except Exception:
            continue
    return None
```

## IP Reservation

### ProxyReservation Table

The `ProxyReservation` table tracks which IPs are currently in use, preventing multiple apps from using the same IP on the same target domain:

```python
class ProxyReservation(Base):
    __tablename__ = "proxy_reservation"

    id: Mapped[int] = mapped_column(primary_key=True, autoincrement=True)
    ip_address: Mapped[str] = mapped_column(String(45))          # IPv4 or IPv6
    target_domain: Mapped[str] = mapped_column(String(255))      # e.g., "gov.de"
    app_id: Mapped[str] = mapped_column(String(64))              # Which app reserved it
    reserved_at: Mapped[datetime] = mapped_column(DateTime)      # When reserved
    ttl_seconds: Mapped[int] = mapped_column(Integer, default=1800)  # 30 min default
    released: Mapped[bool] = mapped_column(Boolean, default=False)   # Explicit release flag
```

**Database constraint — unique partial index:**

```sql
-- PostgreSQL
CREATE UNIQUE INDEX uq_proxy_reservation_ip_address
ON proxy_reservation (ip_address, target_domain)
WHERE released = false;

-- MSSQL (filtered index, same semantics)
CREATE UNIQUE INDEX uq_proxy_reservation_ip_address
ON proxy_reservation (ip_address, target_domain)
WHERE released = 0;
```

This index guarantees that no two active (unreleased) reservations can exist for the same IP + domain combination, regardless of application-level timing. It is the ultimate safety net against race conditions across threads, processes, and servers.

See [1.2. data-model.md](1.2.%20data-model.md) for full table specification.

### Domain Scoping: Reservation vs Blacklist

The proxy system uses two different domain scoping levels:

**ProxyReservation uses root domain** (e.g., `gov.de` from `sub.gov.de`). Reservations prevent IP collision across ALL subdomains of the same site, since target sites typically track IPs globally across subdomains. Root domain extraction uses `tldextract` (Mozilla Public Suffix List) to correctly handle TLD edge cases like `.gov.de`, `.co.uk`, `.com.au`.

**ProxyBlacklist uses full subdomain** (e.g., `sub.gov.de`). A ban on one subdomain does not necessarily imply a ban on other subdomains of the same domain.

`ProxyManager` handles this automatically: the constructor takes a full `target_domain` (e.g., `"sub.gov.de"`) and extracts the root domain internally via `extract_root_domain()`.

### Lock Scope: Root Domain

Reservations are scoped by **root domain** — the registered domain of the website being scraped. Apps hitting any subdomain of the same domain can't use the same IP, regardless of which app group or parent they belong to. This prevents the target site from seeing multiple requests from the same IP that appear to come from different users/sessions.

### Atomic Reservation (Race Condition Safety)

The reservation check and insert must be **atomic**. Without this, two threads (or two separate apps scraping the same domain) could both see an IP as available and both reserve it:

```
Thread A: check IP → available         Thread B: check IP → available
Thread A: INSERT reservation            Thread B: INSERT reservation  ← duplicate!
```

The framework prevents this with a two-layer defense:

**Layer 1 — Atomic `_try_reserve()` method:**

```python
def _try_reserve(self, ip: str) -> bool:
    """Atomic check + reserve in a single transaction.

    Returns True if reservation succeeded, False if IP already taken.
    Works safely across threads, processes, and servers.
    """
    with self.database.create_session() as session:
        # Step 1: Release any TTL-expired reservation for this IP + domain
        session.execute(
            update(ProxyReservation)
            .where(
                ProxyReservation.ip_address == ip,
                ProxyReservation.target_domain == self.target_domain,
                ProxyReservation.released == False,
                func.now() > ProxyReservation.reserved_at + ttl_interval
            )
            .values(released=True)
        )

        # Step 2: Try INSERT — unique index is the arbiter
        try:
            session.add(ProxyReservation(
                ip_address=ip,
                target_domain=self.target_domain,
                app_id=self.app_id,
                reserved_at=datetime.now(),
                ttl_seconds=self.ttl_seconds
            ))
            session.commit()
            return True
        except IntegrityError:
            session.rollback()
            return False  # Another thread/app reserved it first
```

**Layer 2 — Unique partial index catches edge cases:**

Even if two transactions execute the UPDATE + INSERT simultaneously, the `uq_proxy_reservation_ip_address` unique index on `(ip_address, target_domain) WHERE released = false` causes the second INSERT to fail with `IntegrityError`. The framework catches this gracefully and retries with a new proxy session.

**Why both layers?**

| Layer | Purpose | Scope |
|-------|---------|-------|
| Unique partial index | DB-level guarantee — no duplicate active reservations ever | Threads, processes, servers |
| `IntegrityError` catch | Graceful application-level retry when index fires | Acquire loop retries with new session |
| TTL UPDATE in same TX | Cleans expired rows before INSERT attempt | Crash safety within the atomic flow |

**Why no deadlocks?** The pattern uses no `SELECT FOR UPDATE` — no row locks are held across statements. The UPDATE only touches expired rows (nobody else contends for those). The INSERT either succeeds atomically or fails with `IntegrityError` immediately. No blocking, no waiting.

### Race Condition Scenarios

**Scenario 1 — Two threads from the same app get the same IP:**

```
Thread A                                Thread B
────────                                ────────
verify_ip() → "1.2.3.4"                verify_ip() → "1.2.3.4"

_try_reserve("1.2.3.4"):               _try_reserve("1.2.3.4"):
  BEGIN TX                                BEGIN TX
  UPDATE expired → 0 rows                UPDATE expired → 0 rows
  INSERT (1.2.3.4, domain)               INSERT (1.2.3.4, domain)
  COMMIT [SUCCESS]                       ← IntegrityError (unique index)
  return True                             ROLLBACK, return False

→ Uses proxy [SUCCESS]                  → Retries with new session_id
                                        → Gets different IP → reserves [SUCCESS]
```

**Scenario 2 — Two different apps targeting the same domain:**

Identical behavior — the database doesn't care if the competing transactions come from threads or processes. The unique index is the single arbiter.

### TTL-Based Auto-Expiry (Crash Safety)

If an app crashes without releasing its proxy reservation, the row stays in the database. TTL-based expiry handles this automatically — reservations older than their TTL are treated as expired and cleaned up during the next `_try_reserve()` call:

- **Default TTL:** 1800 seconds (30 minutes)
- **How it works:** The `_try_reserve()` UPDATE step releases expired reservations before attempting the INSERT
- **No cleanup job needed for active reservations:** Stale rows are cleaned up organically as part of the acquisition flow
- **Daemon cleanup thread:** Each `ProxyManager` instance runs a daemon thread that periodically deletes its own released reservations past the cooldown period
- **Janitor CLI:** `python -m data_collector.tables proxy-cleanup` deletes all orphaned rows across all apps (released past cooldown + unreleased past TTL from crashes)

This means a crashed app's proxy reservations auto-heal within 30 minutes without any manual intervention. Released reservations are cleaned up by the daemon thread during normal operation and by the janitor for crash orphans.

### Acquisition Flow

```python
class ProxyManager:
    def __init__(self, provider: ProxyProvider, database: Database,
                 target_domain: str, app_id: str,
                 judges: list[str] | None = None,
                 ttl_seconds: int = 1800,
                 cooldown_seconds: int = 300,
                 acquire_timeout: int = 120,
                 recheck_interval: int = 5,
                 cleanup_interval: int | None = None):
        self.provider = provider
        self.database = database
        self.target_domain = target_domain
        self.app_id = app_id
        self.judges = judges or PROXY_JUDGES
        self.ttl_seconds = ttl_seconds
        self.cooldown_seconds = cooldown_seconds
        self.acquire_timeout = acquire_timeout
        self.recheck_interval = recheck_interval
        self.cleanup_interval = cleanup_interval or cooldown_seconds
        # Starts daemon cleanup thread (dies with process)

    def acquire(self, logger) -> Proxy:
        """Acquire a unique, reserved proxy for this domain.

        Returns:
            Proxy object with verified IP, URL dict, session ID, and target domain.

        Raises:
            ProxyAcquisitionTimeout — if no unique proxy found within timeout
        """
        start = datetime.now()
        while (datetime.now() - start).total_seconds() < self.acquire_timeout:
            # 1. Generate new session → provider builds proxy URL
            session_id = uuid.uuid4().hex
            proxy_url = self.provider.build_proxy_url(session_id)

            # 2. Verify IP via proxy judge (with failover)
            ip = self._verify_ip(proxy_url)
            if ip is None:
                logger.warning("All proxy judges failed, retrying with new session")
                continue

            # 3. Atomic reserve — INSERT-or-fail with unique index
            if self._try_reserve(ip):
                logger.info("Proxy acquired", ip=ip, domain=self.target_domain)
                return Proxy(
                    url=proxy_url,
                    ip_address=ip,
                    session_id=session_id,
                    target_domain=self.target_domain
                )

            # IP already reserved by another thread/app — retry with new session
            logger.debug("IP already reserved", ip=ip, domain=self.target_domain)
            sleep(self.recheck_interval)

        raise ProxyAcquisitionTimeout(
            f"No unique proxy acquired for {self.target_domain} in {self.acquire_timeout}s"
        )

    def release(self, ip_address: str):
        """Release proxy reservation — call on thread/session completion."""
        ...

    def report_failure(self, ip_address: str):
        """Record a proxy failure for blacklist tracking.

        Called by scraper code when target requests fail with proxy-related
        errors (403, CAPTCHA, connection refused). Escalates the lockout
        level for this IP on the target subdomain.
        """
        ...

    def cleanup_reservations(self) -> int:
        """Delete own released reservations past the cooldown period.

        Scoped to self.app_id. Called by daemon thread periodically.
        Returns number of deleted rows.
        """
        ...

    def shutdown(self):
        """Flush all released reservations for this app and stop the daemon thread.

        Called during app cleanup. Deletes all released rows for this app_id
        regardless of cooldown. Use stop() for lightweight thread-only shutdown.
        """
        ...

    def stop(self):
        """Signal the daemon cleanup thread to stop gracefully."""
        ...
```

### Cooldown

Released IPs cannot be reused within `cooldown_seconds` (default 300 = 5 minutes). This prevents rapid IP reuse across sibling apps in the same parent, reducing bot detection risk. The cooldown check runs inside `_try_reserve()` before the INSERT.

### Reservation Cleanup

Three cleanup mechanisms prevent unbounded table growth:

**App shutdown** -- Call ``shutdown()`` in the scraper's ``cleanup()`` method. This deletes all released rows for this ``app_id`` regardless of cooldown (the app is done, its own cooldown is irrelevant) and stops the daemon thread. Sibling apps are still protected by the cooldown check in ``_try_reserve()``.

**Daemon cleanup thread** -- Each `ProxyManager` starts a daemon thread that periodically (every `cleanup_interval` seconds) deletes its own `released=True` rows past the cooldown period. The thread is scoped to `self.app_id` and dies automatically when the process exits. Handles cleanup of long-running apps where proxies are released mid-run.

**Janitor CLI** -- Deletes all orphaned rows across all apps:

```bash
python -m data_collector.tables proxy-cleanup                          # Defaults: cooldown=300, ttl=1800
python -m data_collector.tables proxy-cleanup --cooldown 600 --ttl 3600  # Custom thresholds
```

The janitor deletes two categories of dead rows:
- `released=True` rows older than `--cooldown` seconds (past cooldown, any app)
- `released=False` rows older than `--ttl` seconds (crash orphans from force-killed processes)

Safe to run while apps are active. The Manager (WP-09) will call the janitor on startup and periodically.

## IP Blacklist

### ProxyBlacklist Table

The `ProxyBlacklist` table tracks problematic proxy IPs per subdomain with incremental lockout:

```python
class ProxyBlacklist(Base):
    __tablename__ = "proxy_blacklist"

    id = auto_increment_column()
    ip_address = Column(String(45), nullable=False)
    target_domain = Column(String(255), nullable=False)    # Full subdomain
    failure_count = Column(Integer, default=1, nullable=False)
    first_failure_at = Column(DateTime, nullable=False)
    last_failure_at = Column(DateTime, nullable=False)
    lockout_until = Column(DateTime, nullable=True)
    lockout_level = Column(Integer, default=0, nullable=False)
    is_banned = Column(Boolean, default=False, nullable=False)
```

**Unique constraint:** `(ip_address, target_domain)` prevents duplicate entries.

### Lockout Progression

Each failure escalates the lockout duration. After exceeding the maximum level, the IP is permanently banned for that subdomain:

| Level | Duration |
|-------|----------|
| 0 | 5 minutes |
| 1 | 20 minutes |
| 2 | 1 hour |
| 3 | 4 hours |
| 4 | 24 hours |
| 5+ | Permanent ban (`is_banned = True`) |

Lockout durations are configurable via the `lockout_durations` parameter on `ProxyManager`.

### BlacklistChecker

`BlacklistChecker` manages the blacklist with three operations:

- **`is_locked_out(ip_address)`** -- checks if an IP has an active lockout or permanent ban for this subdomain. Called during proxy acquisition to skip known-bad IPs.
- **`record_failure(ip_address)`** -- upserts a failure record, increments the failure count, and escalates the lockout level. Called via `ProxyManager.report_failure()` when scraper requests fail.
- **`cleanup_expired()`** -- deletes non-banned entries older than the retention period (default 30 days). Prevents the blacklist table from growing unboundedly.

### Retention Strategy

To prevent table bloat from millions of proxy IPs:

- Only IPs that have been reserved AND reported as failed are recorded (not all generated IPs)
- Non-banned entries are automatically cleaned up after `retention_days` (default 30)
- Permanently banned entries persist until manually removed
- Cleanup runs on demand via `cleanup_expired()` or periodically via a scheduled job

## Ban Detection

| Signal | Action |
|--------|--------|
| HTTP 403 | Rotate proxy (release + acquire new) |
| HTTP 429 | Cooldown, retry with delay |
| CAPTCHA page | Route to captcha solver ([12. captcha.md](12.%20captcha.md)) |
| Connection refused | Mark proxy as failed, acquire new |
| Timeout | Retry once, then acquire new |

Ban detection integrates with the Request class error introspection methods (`is_blocked()`, `is_proxy_error()`) and `RequestMetrics` target health circuit breaker — see [4.4. request.md](4.4.%20request.md).

## Multi-Threaded Proxy Usage

Each thread acquires its own unique proxy via `ProxyManager` and releases it on completion. `RequestMetrics` (shared across threads) tracks per-proxy performance:

```python
from data_collector.utilities.request import Request, RequestMetrics
from data_collector.proxy import ProxyManager, BrightDataProvider

class CourtCases(BaseScraper):
    def collect(self):
        with ThreadPoolExecutor(max_workers=5) as executor:
            futures = {
                executor.submit(self.collect_detail, case, idx): case
                for idx, case in enumerate(self.work_list)
            }
            for future in as_completed(futures):
                if self.fatal_flag:
                    break
                future.result()

    def collect_detail(self, case, instance_id):
        """Per-thread: acquire proxy → fetch → parse → store → release."""
        proxy_mgr = ProxyManager(
            provider=self.proxy_provider,
            database=self.database,
            target_domain="example.com",
            app_id=self.app_id
        )
        proxy = proxy_mgr.acquire(self.logger)

        try:
            req = Request(timeout=120, retries=3, metrics=self.metrics)
            req.set_proxy(proxy.url)
            req.set_headers({"User-Agent": "Mozilla/5.0 ..."})

            response = req.get(f"https://example.com/case/{case.case_id}")

            if req.should_abort(self.logger, proxy_on=True):
                return

            data = self.parser.parse_case_detail(response.content)
            self.store([data])
            self.solved += 1
        finally:
            proxy_mgr.release(proxy.ip_address)  # Always release
```

After all threads complete, `metrics.log_stats()` includes per-proxy breakdown (success rate, latency) identified by `country:port` — no credentials logged.

`ProxySettings` provides zone-parameterized environment variable loading for proxy credentials. Each BrightData zone (mapped to an `app_parent`) has its own set of env vars following the pattern `DC_PROXY_{FIELD}_{ZONE}`:

| Env Var Pattern | Required | Default | Description |
|----------------|----------|---------|-------------|
| `DC_PROXY_HOST_{ZONE}` | Yes | -- | Gateway hostname (e.g., `brd.superproxy.io`) |
| `DC_PROXY_PORT_{ZONE}` | Yes | -- | Gateway port (e.g., `22225`) |
| `DC_PROXY_USERNAME_{ZONE}` | Yes | -- | Provider username (includes BrightData zone) |
| `DC_PROXY_PASSWORD_{ZONE}` | Yes | -- | Provider password |
| `DC_PROXY_COUNTRY_{ZONE}` | No | `None` | ISO country code for geo-targeting |
| `DC_PROXY_PROTOCOL_{ZONE}` | No | `http` | Proxy protocol (`http`, `https`, `socks5`) |

The zone suffix is the uppercase app_parent name. For example, apps under `data_collector/examples/scraping/` use zone `SCRAPING`, while apps under `data_collector/croatia/registry/` use zone `REGISTRY`. This design enables per-zone expense tracking on the BrightData dashboard.

```python
from data_collector.settings.proxy import ProxySettings
from data_collector.proxy import BrightDataProvider, ProxyManager

# Load zone-specific credentials from environment
proxy_data = ProxySettings.from_zone("scraping").to_proxy_data()
provider = BrightDataProvider(proxy_data)
proxy_manager = ProxyManager(provider=provider, database=database, target_domain="example.com", app_id=app_id)
```

**Module path:** `data_collector/settings/proxy.py`

## Example Applications

Two example applications demonstrate proxy integration with the scraping framework. Both reuse existing parsers and tables from the books and quotes_authors examples.

### Single-Threaded (proxy_single)

**Module path:** `data_collector/examples/scraping/proxy_single/main.py`

Acquires one proxy at the start of `collect()`, uses it for all page requests via `Request.set_proxy()`, and releases it in `cleanup()`. Proxy failures trigger `ProxyManager.report_failure()` for blacklist escalation.

Run: `python -m data_collector.examples run scraping/proxy_single/main`

### Multi-Threaded with Per-Item Rotation (proxy_threaded)

**Module path:** `data_collector/examples/scraping/proxy_threaded/main.py`

Each worker thread acquires a fresh proxy IP per work item via `ProxyManager.acquire()`, makes one request, then releases the proxy in a `finally` block. This per-item rotation prevents IP reuse across sequential requests, which is the primary bot detection signal on target sites.

The `ProxyManager`'s atomic database reservation prevents two concurrent workers from receiving the same IP for the same domain. After release, the IP returns to the pool with TTL + blacklist cooldowns preventing immediate reuse.

Run: `python -m data_collector.examples run scraping/proxy_threaded/main`
